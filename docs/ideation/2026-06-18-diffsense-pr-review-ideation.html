diffsense — reviewing AI code at AI speed

Seven ranked directions for a card-based, AI-assisted PR review tool built for the new bottleneck: not writing code, but reviewing the mountains of it that agents now produce. The set is chosen to be fast and playful without becoming a rubber-stamp machine.

Topic Context

The bottleneck is real and measured. On high-AI-adoption teams, PR size is up 154% and review time up 91%, yet company-level delivery metrics stay flat — the gain from faster generation is eaten by slower review (Faros AI). Defect detection collapses with size: 87% of defects caught in PRs under 100 lines, 28% at 1,000+ lines; an XL PR gets ~1.8 comments in 4.2 hours of "review" (Propel Code). And 61% of AI-generated PRs receive zero recorded review activity — rubber-stamping by omission (arXiv 2605.02273).

The whitespace is the reviewer's UI. Every major tool — CodeRabbit, Greptile, Graphite Agent, Qodo, GitHub Copilot review, Korbit, Cursor BugBot — ships the same two-layer paradigm: a top-level AI summary plus inline comments on diff lines. None use a card/triage queue with per-chunk disposition. CodeRabbit's "Atlas" (2025) is the closest: change-cohorts, layer navigation, J/K keys, rendered diagrams — but it has no per-chunk risk score, no swipe disposition, no reviewer-driven chunk splitting, and no global score. Graphite owns the PR author workflow; nobody owns the PR reviewer workflow with a purpose-built UI. That is the wedge.

Two landmines the design must dodge. First, swipe fatigue: Tinder/Avec-style swiping works for low-stakes, recoverable decisions; a rubber-stamped production bug is neither, so the gesture model has to encode that asymmetry. Second, opaque scoring: in security triage (Snyk, Apiiro) engineers disengage from a single number they can't interrogate — multi-factor scores that show their breakdown are trusted; bare scalars are ignored. The strongest analogy across the research is the radiology reading queue: a worklist of studies, each with an AI pre-read and a priority flag, signed off one at a time — AI triage as attention-direction, not judgment-replacement.

Feasibility. Logical-chunk decomposition is tractable via AST / call-graph / import-graph cohorting (SemanticDiff already groups renames into one card and detects moved code); the hard part is the "reviewable unit" heuristic — too fine yields hundreds of trivial cards, too coarse collapses back to monolithic file review. Structural signals (lines added/deleted, files, entropy) predict review effort at AUC 0.96 at PR creation time, before a human opens anything — a strong, cheap basis for ordering and gating.

1.

Risk-Ranked Queue with an Attention Budget

A5 · scale & workflowConfidence 85%Complexity Medium

Stop presenting chunks in file order. At PR load, order the deck by predicted risk using creation-time structural signals (lines ±, files touched, API-boundary crossing, test-coverage delta), riskiest first while attention is freshest. The reviewer sets an explicit attention budget — "20 minutes" or "15 cards." When it's spent, the remaining low-risk tail isn't silently skipped: it collapses into one honest, logged "sampled-accept" card the reviewer confirms with eyes open. This is the spine the other six ideas plug into — the radiology worklist made literal.

Basis
external: arXiv 2601.00753 — structural signals predict review effort at AUC 0.96 at PR creation (semantic/CodeBERT only 0.52); radiology reading-queue model (AI-positive studies promoted, hardest cases seen first); Propel Code — XL PRs get 1.8 comments because late-queue items get skimmed.

Why it mattersIt converts the XL-PR problem from "too much to read" into "spend your real attention on the right 15%." It's also buildable on day one with only structural signals — no AI explanation needed to validate the ordering — so it de-risks the whole product.

Downsides
Risk ranking that's wrong erodes trust fast — a "safe" chunk that ships a bug is worse than a missed one in a flat list. The auto-collapsed tail is where liability lives; the sampled-accept needs a crisp audit trail and an easy "expand it all" escape hatch.
2.

Intent as the Unit of Review

A1 · decompositionConfidence 80%Complexity Medium-High

The "PR" is an arbitrary packaging boundary — usually just whatever an agent did in one run. The semantically meaningful unit is the intent ("add rate limiting to the API"). Group cards under declared intents, and — most valuably — flag every diff region that maps to no declared intent as scope creep. Unrequested, drive-by changes are the single highest-risk content in agent PRs, and they're exactly what a line-by-line reviewer skims past. This also gives the global score a natural decomposition (coverage per intent) instead of a line-weighted average, and it makes the "reviewable unit" heuristic tractable by anchoring chunks to a known target rather than inferring boundaries from AST alone.

Basis
reasoned: Agent PRs are generated from a prompt/spec, so a machine-readable intent already exists upstream of the diff. A PR groups "whatever the agent changed," not "what was requested" — so the gap between the two (undeclared edits) is both detectable and the most dangerous content. No competitor isolates it. external: Faros — agent PRs are larger and driftier; the scope-creep surface grows with PR size.

Why it mattersThis is the most AI-native idea in the set — it only makes sense in a world where code comes from an agent with a stated goal, and it attacks the failure mode (silent scope creep) that human-oriented review tools structurally miss. It's a differentiation story competitors can't bolt on.

Downsides
Depends on a declared intent existing and being trustworthy. For human PRs or terse "fix stuff" descriptions you must infer intent (an LLM step that can be wrong), and mis-mapping a legitimate change as "scope creep" is an annoying false positive. Strongest when wired directly to the coding agent that emits structured intents.
3.

Disposition-by-Defense — a Swipe That Costs More When It Should

A2 · triage interactionConfidence 85%Complexity Low-Medium

The swipe is the product's biggest moral hazard: it makes approval frictionless on exactly the PRs that already get rubber-stamped. Fix it with variable friction. Trivial chunks (renames, formatting, covered-by-tests) stay one-keystroke fast — that's the speed promise. But to accept a high-risk chunk, the reviewer must clear a one-tap, AI-generated comprehension check or "time-out" ("This widens auth scope from 24h to 1h token expiry — confirm you traced the call sites: y/n"). Friction becomes a feature, applied surgically where mindless approval is most expensive. The check pass-rate is itself a real, sellable anti-rubber-stamp metric.

Basis
external: Tinder/Avec swipe failure mode is "swipe fatigue → mindless throughput"; the WHO surgical "time-out" is the canonical mechanism for deliberately interrupting a fluent expert before an irreversible act. reasoned: matching gesture cost to stakes is the one lever that lets the product be both fast and honestly claim higher quality.

Why it mattersIt directly resolves the central tension the user named — playful/fast vs. highly professional. Without it, a swipe UI risks industrializing rubber-stamping; with it, the friction is the proof of diligence and the source of the quality metric.

Downsides
Friction is friction — set the risk threshold too low and reviewers feel nagged and route around it; too high and it never fires. The comprehension question must be genuinely answerable from the card, or it becomes a "click yes" reflex (rubber-stamping the gate itself).
4.

A Risk Portfolio, Not a Number

A4 · scoringConfidence 85%Complexity Medium

The global "score" is the documented trust-killer when it's a bare scalar. Replace the single number with a portfolio: a small set of named, evidenced risk positions — "2 untested API-boundary changes," "1 unrequested data-model edit," "auth scope widened" — each linking to the chunks that created it. The score becomes the shape of the risk, not a digit. And make it live: when a reviewer disagrees with the AI's read on a chunk and overrides it, the portfolio recomputes in real time, turning the score from an AI verdict into a shared reference point the team negotiates against. You clear positions; you don't chase a number.

Basis
external: Snyk / Apiiro — engineers disengage from opaque scores they can't interrogate; multi-factor scores that expose their breakdown are trusted; best practice is advisory mode before enforcement. A negotiable, override-able score is inherently advisory — the human stays the authority.

Why it mattersThis is the trust foundation for everything that touches scoring — including any future merge-gate. Every scoring competitor lost credibility with scalars; a portfolio sidesteps the exact landmine the research names, and the override history is a clean training signal for where the AI's calibration drifts from the team's.

Downsides
Teams and dashboards want a single number to sort and gate on; a portfolio resists that and needs a defensible roll-up anyway. Defining the canonical set of "risk position" types is real product work, and a portfolio with 15 positions is its own kind of noise.
5.

Falsifiable AI Claims, One-Tap Refutable

A3 · explanation qualityConfidence 75%Complexity Medium

Don't render the AI's explanation as trust-me prose ("This updates the session token"). Render it as a short set of falsifiable claims: [CLAIM] does not change the public API contract · [CLAIM] the test on line 88 covers the new branch. Each claim is one-tap "wrong," which (a) instantly expands to the diff proving or disproving it, and (b) logs a specific calibration signal. LLMs hallucinate most dangerously in prose — confidently asserting a test covers a branch it doesn't — and prose hides it. A claim forces the model to commit to something checkable, and forces diffsense to back it with evidence.

Basis
external: the same Snyk principle as the score — decompose the opaque assertion into interrogable factors. reasoned: a falsifiable claim is the textual equivalent of a score's factor-breakdown; "mark wrong" yields structured, collectable calibration data ("we mis-assess async test coverage 23% of the time") where freeform prose feedback is unstructured and lost.

Why it mattersIt raises the AI-explanation bar from "plausible" to "accountable," which is the only way a reviewer learns when to trust the AI versus look themselves. The refutation stream is the highest-quality training data the product can collect — revealed reviewer judgment, not survey clicks.

Downsides
Forcing claims into a fixed schema can flatten nuance the prose captured, and a confidently-wrong claim is more dangerous than vague prose if reviewers trust the format. Enumerating the claim-set that covers ~80% of review concerns is unproven research work.
6.

Reviewability Gate — PR Admissions Control

A5 · prevention & workflowConfidence 70%Complexity Medium-High

The highest-leverage intervention happens before a human is ever paged. Run diffsense as a pre-merge gate that refuses PRs whose creation-time signals predict zero effective review — enormous size, no test delta, heavy API-boundary churn, no declared intent. The author (often the agent itself) must split it, annotate it, or justify it before a reviewer is summoned. This stops competing on "nicer summary of an unreviewable PR" and instead attacks the actual failure mode — rubber-stamping by omission — by refusing to let unreviewable PRs exist. It also moves the budget line from "nice-to-have tool" to CI/policy, a moment nobody currently owns.

Basis
external: arXiv 2601.00753 — the strongest predictive signal (AUC 0.96) exists at PR creation time, so the highest-leverage move is pre-review, not during; arXiv 2605.02273 — 61% of agent PRs get zero review, i.e. the worst PRs are exactly the unreviewable ones a gate would catch.

Why it mattersIt's a defensible wedge that converts diffsense from a viewer into infrastructure — "PR admissions control." Pairs naturally with idea 1 (the gate uses the same risk model that orders the queue) and idea 2 (a PR with no declared intent is, by definition, not admissible).

Downsides
Blocking developers is the fastest way to get a tool uninstalled — enforcement backlash is the documented Snyk failure mode, so this must launch in advisory mode and earn trust before it blocks. Legitimate large PRs (vendored deps, generated code, big refactors) need clean override paths or it becomes the thing everyone routes around.

7.

Chunk Fingerprint Registry — Review Memory That Compounds

A4 · compounding memoryConfidence 70%Complexity High

Every chunk that passes through diffsense gets a structural fingerprint (AST shape, call-graph position, file-type signature, test-coverage delta). When a similar pattern recurs in a later PR — same repo or across the org — surface the prior disposition: "Chunks like this were accepted 12/14 times here; the 2 rejections were both in auth/." Not a rule engine — a precedent-recall surface. And every time a reviewer splits a chunk (the git-add-p gesture), record the AST boundary they chose; aggregate enough and the decomposition model learns where humans want boundaries, directly attacking the hardest open problem (the "reviewable unit" heuristic). More reviews → sharper chunks and sharper precedent → faster reviews. That loop is the moat.

Basis
external: SemanticDiff already fingerprints structure (groups renames into one card, detects moved code) — the infrastructure exists. reasoned: human split decisions are ground-truth labels for "where a reviewable unit belongs" — a free supervision signal no static heuristic can match; precedent recall directly counters AI-generated boilerplate that repeats across PRs.

Why it mattersIt's the durable defensibility story. A chunking model trained on 10,000 human split decisions in a specific stack — plus org-level precedent — is something a well-funded competitor can't copy without the same review history. It turns usage into a widening lead.

Downsides
Severe cold-start — worthless on day one and on every new repo until volume accumulates, so it can't be the launch wedge. Precedent recall can ossify bad habits ("we always approved this, so approve it") and the fingerprint-similarity matching is genuine ML engineering, not a weekend feature.
Two big swings parked for later (not cut on merit)

Review the generation, not the diff — hook into the agent loop so risky chunks are caught and regenerated before a PR ever opens, shrinking the artifact at birth rather than coping with its size. Highest ceiling in the whole set, but it shifts the product from "PR reviewer" toward "agent-loop guardrail," so it's a deliberate phase-2 bet. And call-graph blast-radius edges between cards — show that approving chunk 12 (a signature change) implicates chunks 31, 47, 89; reverse 12 and they re-open. Novel and grounded (Apiiro Risk Graph), parked only to keep the launch set focused.
